Section 01

The Problem

We built a benchmark to compare 10 verification methods across 11 tasks with 44 seeded bugs. Then we ran it. The result was a flatline.

Differentiating bugs

Out of 44 seeded bugs, exactly one (ALG-1 bug3) differentiated any method from any other. Eight methods scored an identical 36/44. The remaining two scored 35/44. The entire benchmark reduced to a coin flip on a single duplicate-index edge case.

The user's actual goal is not "which method wins on these synthetic tasks?" but "how do I build an agent that writes code without bugs, can map all relevant information, and iterate?"

— DRR-2026-04-06, Problem Statement

Six Root Causes of Saturation

1. Static Comparison

Methods are pinned prompts, not configurable agents. No room to iterate or improve.

2. Shared Test Suites

8/11 tasks use identical tests across all methods — no per-method differentiation possible.

3. One-Shot Runs

No iteration loop. Agents cannot recover from their own errors.

4. Minimal Doc Bootstrapping

Methods relying on Effect, XState, Dafny, Lean, Coq, fast-check fail on toolchain quirks their training set does not teach.

5. Synthetic Textbook Tasks

All 11 tasks are canonical problems in LLM training sets — binary search, LRU cache, RBAC.

6. Hand-Seeded Bugs

44 total, 18% universally undetectable. The ceiling is artificial.

Section 02

The Hypothesis

Methods are not the right unit of comparison. Agents are.

A "method" (Effect TS, TDD, Dafny, etc.) is a static label. It says nothing about the docs given to the model, the tools it can call, or whether it can iterate on failures. When we say "Effect TS failed," we really mean "the first version of an Effect TS agent, with no relevant docs and no feedback loop, failed."

The Four-Axis Agent Identity

An agent is fully described by four axes. Model is orthogonal (runtime parameter).

Method × Version × Tools × Context Bundle

effect-ts/v2 = Effect TS + v2 + [typecheck, vitest] + effect-quickstart-bundle

H1: "Effect TS Is Not Bad — It Just Needs Better Docs"

During calibration, Effect TS (method B) and PBT+Effect (method I) needed retries to pass. Plain TS (method A) passed first-try. The failure mode was always the same:

FiberFailure: "The fiber was terminated with an unhandled error"
  // Effect wraps all exceptions in FiberFailure.
  // Without docs, the model doesn't know to unwrap or .pipe(Effect.catchAll(...))
  // The generated code compiles but tests throw opaque wrappers.

Hypothesis H1

effect-ts/v2 (v1 + Effect quickstart + FiberFailure pitfalls bundle) will score ≥ plain-ts/v1 on BL-1 and BL-5 on Haiku one-shot, whereas effect-ts/v1 scored below plain-ts/v1 due to FiberFailure wrapping bugs.

Hypothesis H2

Adding a typecheck loop (effect-ts/v3 = v2 + typecheck) will recover at least 50% of the remaining delta vs plain-ts/v1 baseline.

Hypothesis H3

On the hardened tier (Phase 9 tasks), methods with iteration loops will beat their non-iterating counterparts by ≥ 10 prevention points absolute. If false, loop infrastructure is not justified for real-repo tasks.

The Agent Runner

The runner (infra/agent-runner.mts) is a generalization of the Phase 6.5 audit harness. It:

  1. Loads a manifest YAML — method, version, tools, context bundle path
  2. Injects the context bundle into systemPrompt
  3. Passes tool declarations to the Claude Agent SDK
  4. Runs the agent against a task's spec.md
  5. If loops are configured: feeds typecheck/vitest output back as user messages for multi-turn runs
  6. Records full trajectory JSON: tokens, turns, wall time, code, test results

Context Bundles

Each bundle is a distilled reference (≤2,000 tokens) extracted from official documentation. Versioned with upstream hashes so we know when the source changes.

The Surprise

Failures are API shape mismatch, not algorithmic difficulty

Five of the seven hard tasks collapsed because Haiku read an ambiguous spec and picked a different export style than the tests expect — class vs. factory, toposort vs. topoSort, prorateCost vs. calculateProration. The core algorithms were often correct; the wiring was wrong.

This is a legitimate signal of difficulty. Matching a spec precisely without iterating is a real capability. Methods with test-feedback loops (TDD), formal specs (Dafny/Lean/Coq), or iterative reasoning (Consensus) are designed to recover from exactly this kind of miss.

Implication

Agents with iteration loops will recover these points. The gap between a bare Haiku one-shot (62.7%) and a looped agent represents the measurable value of agent infrastructure — docs, tools, and feedback.

Contamination Analysis

Claude Opus 4.6 training cutoff is approximately May 2025. Any benchmark with tasks predating this is potentially contaminated.

Gold Standard (post-cutoff)

Web-Bench (May 2025), SWE-bench Live (rolling), SEC-Bench (Jun 2025). Use these as primary differentiators.

Borderline (near cutoff)

Multi-SWE-bench, SWE-Lancer, SWE-PolyBench. Issues pre-date cutoff but test harnesses are fresh. Treat with caution.

Contaminated (use as floor only)

BugsJS, Defects4J, EvalPlus, HumanEval, APPS, CodeContests. Useful for porting mechanics and sanity checking, not for ranking methods.

Strategy: build the "hardened" slice from Web-Bench, SWE-bench Live latest, LiveCodeBench ≥ Jun 2025 filter, SEC-Bench, and SWE-Lancer Diamond. Keep the older benches as a "sanity floor."

The FPF Decision

Seven solution directions were brainstormed, scored on 9 weighted dimensions, and analyzed for weak links. One emerged dominant.

The 9 Evaluation Dimensions

#DimensionWeightWhat It Measures
1Discrimination powerHighBits of score variance across methods
2Prevention measurabilityHighBugs escaped / mutations survived
3Iteration captureHighTurns / feedback loops supported
4Context-mapping captureHighRetrieval recall/precision measurability
5RealismMediumFraction of tasks from real repos
6Contamination resistanceMediumFraction post-training-cutoff
7Economic rankingMedium$/bug prevented or tokens/pass
8ReproducibilityMediumSame config → same result
9Implementation costLowWeeks of refactor

Seven Solution Variants

S1 Keep current benchmark, add harder tasks 21/45

Minimal refactor. Still does not capture iteration or docs uplift; does not improve discrimination fundamentally.

S2 Compose methods into a single stack 21/45

Replace "A vs B vs C" with "A + B + C pipeline." Directly targets "bug-free" via defense-in-depth. Loses per-method attribution.

S3 Agent loop wrapper around existing methods 27/45

Add typecheck+test feedback loop. Cheap. Captures iteration. Does not address docs bootstrapping or versioning.

S4 Four-axis versioned agent framework 37/45

Methods become (method × version × tools × context_bundle) configurations. Model is runtime axis. Tournament over all configs. Captures all three user goals. Every improvement is a versioned artifact with measurable delta.

S5 Pure mutation testing benchmark 26/45

Replace hand-seeded bugs with Stryker auto-mutations. Eliminates human bias. Does not capture generation ability.

S6 SWE-bench-only rebuild 33/45

Drop synthetic tasks, rebuild on SWE-PolyBench + SWE-bench Live. Maximal realism. Loses calibration reference; formal methods cannot operate on real-repo tasks.

S7 Self-improving agent loop (Karpathy autoresearch) 33/45

Time-boxed loop: agent modifies its own prompt/tools, measures scalar metric, iterates. Requires S4 as prerequisite.

Weak-Link Analysis

Why S4 Wins

S4's weakest link is implementation cost (2/5). All other variants have weak links in the high-weight dimensions (discrimination, prevention, iteration, context) — which are the user's explicit goals. By FPF weakest-link assurance, S4 wins because its weak link is in a low-weight dimension.

Congruence

S4 is additive to all other options:

  • S1 (harder tasks) → slot into the framework as task sources
  • S3 (loops) → become one axis of the agent identity
  • S5 (mutation testing) → become a verification oracle
  • S6 (SWE-bench tasks) → become the "hardened tier" task set
  • S2 (stacking) → expressible as a multi-layer combo manifest
  • S7 (self-improving) → expressible as a future self-modifying agent
The implementation cost is front-loaded in Phase 7a (one runner file, ~150 LoC extension of an already-working harness) but amortized across all subsequent phases.

— DRR-2026-04-06, Rationale

What the improver discovered

The winning prompt (iter 5) added 5 structural improvements that no human engineer specified:

  1. Export name matching rules — explicitly name every export to match test import expectations
  2. Validation vs capping distinction — when spec says "capped", clamp; when "invalid", throw
  3. Factory function guidance — createX() must return objects with all methods tests call
  4. Computation correctness hints — percentage discount application order, date proration formulas
  5. Structured pre-coding checklist — imports/exports, edge cases with cap-vs-throw classification

These are "meta-cognitive" improvements — the agent thinks about how to approach the task, not just what to code.

The Karpathy loop: an editable asset, a scalar metric, and a time-boxed cycle.

Editable Prompt → Pass Rate Score → 10-min Cycle → Keep Best

Not monotonic — iterations 1, 3, 4 regressed and were reverted. Exploration has failures. But the best-so-far always advances.

Section 09

Key Discoveries

Five findings from the overnight run that reshape how we think about agent quality.

Discovery 1

“Edge-case identification” beats complexity

plain-ts/v2 wins R1 with a single prompt addition: “list edge cases before coding.” It beats self-critic loops, stack combinations (types + PBT + self-review), and property-based testing agents. The simplest structural change — forcing the model to enumerate edge cases up front — outperforms every elaborate framework.

Discovery 2

H1 Falsified: docs alone don’t fix Effect

The hypothesis that an Effect quickstart docs bundle would eliminate first-try failures was wrong. The real problem is structural interface mismatch: Effect agents produce Effect-typed exports, but test harnesses expect plain TypeScript. The fix is boundary wrapping (v3), not documentation (v2). 12 runs, zero improvement.

Discovery 3

Sonnet overthinks on one-shot

Haiku outperforms Sonnet on one-shot generation: 49% vs 31% average. Sonnet generates more complex abstractions that don’t match test interfaces, uses 2× the tokens, and takes 2× longer. Bigger models need iteration to shine. Without a feedback loop, their extra reasoning capacity becomes a liability.

Discovery 4

Autoresearch is the #1 improvement lever

+14% improvement via AI-driven prompt optimization. The Karpathy loop discovered 5 structural prompt improvements no human specified: export naming rules, validation-vs-capping distinction, factory function patterns, computation hints, and a pre-coding checklist. This is more impactful than docs bundles (H1 falsified), method paradigm choice, or model tier selection.

Discovery 5

BL-5 and SAAS-3 are the frontier

0% for every tournament agent. BL-5 (escrow with ratio-based release) and SAAS-3 (ledger with event sourcing) defeat all agents in R1. The only non-zero scores came from autoresearch iteration 5 (BL-5: 50%) and effect-ts v3 boundary wrapping (SAAS-3: 50%). These tasks are the real test of agent quality — they separate tournament winners from genuine capability.

The Overnight Equation

Edge-case prompt + Autoresearch loop + Cheap model (Haiku) = Best agent

Not the fanciest method. Not the biggest model. The best agent is the one that iterates on its own prompt with a clear scalar signal.
